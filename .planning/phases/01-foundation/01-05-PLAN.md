<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Server Action (addItem / toggleItemActive) | Untrusted form input (item name, category, target id, active flag) crosses into a mutation. |
| Authenticated session → item mutation | A request must carry a valid jose session cookie to mutate the item master. |
| App layer → history-safety invariant | Retire-not-delete (D-09) protects recorded POP line-item snapshots (D-08) from being orphaned. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05-01 | Tampering | addItem input (name/category) | mitigate | Zod schema validates at action entry: `name` is `z.string().trim().min(1)` (rejects empty/whitespace), `category` optional trimmed. Invalid input inserts nothing. |
| T-01-05-02 | Spoofing / Elevation of Privilege | Both Server Actions invoked without auth | mitigate | Each action re-verifies the jose cookie (`verifySession`) before mutating — defense-in-depth beyond proxy.ts (CVE-2025-29927). Unauthorized → throw/redirect, no mutation. |
| T-01-05-03 | Tampering (data integrity) | Hard-deleting an item that has historical POP snapshots | mitigate | Retire is `active=false` ONLY; the scope guard + acceptance gate forbid any `.delete(`/`DELETE FROM item_master`, and the live smoke asserts the row survives a retire. History (D-08 name snapshots) stays intact. |
| T-01-05-04 | Tampering | toggleItemActive(id) targeting a non-existent id | accept | `WHERE id=$1` matches zero rows → no-op; no cross-tenant data (single shared tenant). Low risk; not worth a pre-check round-trip. |
| T-01-05-05 | Information Disclosure | Item list leaking to client | accept | Item names/categories are non-sensitive business metadata, intended to be listed in the UI; no secrets/PII. No `process.env` secret read in any new file. Low risk. |
| T-01-05-SC | Tampering | npm installs | mitigate | This plan installs NOTHING (native inputs + Tailwind only). Acceptance gates assert `package.json` dependencies are unchanged across all three tasks. |

No HIGH-severity threats. Input validation (Zod), auth re-check, and the soft-delete (no-hard-DELETE) invariant are the load-bearing mitigations, each proven by a test or the smoke.
</threat_model>
