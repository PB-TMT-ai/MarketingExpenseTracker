<objective>
Deliver the COMP-04 BACKEND: a deliberate, audited Server Action to record an off-plan-exception execution, plus the re-upload preservation guard that keeps exception rows alive.

This plan delivers:
- `insertExceptionPlanRow(tx, {...})` in plan-rows.ts — inserts a plan_row with `source='exception'`, `exception_reason`, `created_via='actuals-exception'`, returning the new id.
- `addOffPlanExecution` Server Action in executions.ts (alongside saveExecutionsBatch) — `requireSession()` first, Zod-validate (sfid/dealer/who-where/reason required + activity fields), ONE `db.transaction` inserting the exception plan_row THEN the execution FK'd to it, with server trust-recompute (`applyServerCalc`).
- `isUniqueViolation` (23505) catch returning a clean dupe-SFID message (R3 — never a 500).
- The R4 CRITICAL cross-phase guard in plans.ts: scope `commitPlanUpload`'s merge-delete to `source='plan-upload'` so a re-upload never deletes exception rows.
- Action unit tests (happy path, dupe-SFID, tx rollback) + a re-upload preservation regression test.

Purpose: This is the server-side structural guarantee that makes COMP-04 safe — the off-plan guard (COMP-01) is NOT weakened (executions still has no sfid; the exception path inserts a plan_row first then FKs the execution). Plan 05 (the modal + button) calls this action.
