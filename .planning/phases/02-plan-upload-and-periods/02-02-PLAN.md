FK-restrict error detector (RESEARCH §3, §8 — verbatim required pattern):
```
function isFkRestrictError(err: unknown): boolean {
  if (!(err instanceof Error)) return false;
  // Drizzle wraps in DrizzleQueryError → cause is the driver error.
  // PGlite obfuscates its DatabaseError class (electric-sql/pglite#333) so we duck-type on .code.
  const cause = (err as { cause?: unknown }).cause;
  const code = (cause as { code?: string } | undefined)?.code
    ?? (err as { code?: string }).code;
  return code === "23001" || code === "23503";  // 23001 = restrict_violation; 23503 = foreign_key_violation
}
```

<atomic_commit_pattern>
The action body MUST follow exactly this skeleton (RESEARCH §3 is canonical):

```
try {
  const result = await db.transaction(async (tx) => {
    // 1. Snapshot existing rows for (periodId, activity).
    const existing = await tx.select({ id, sfid }).from(planRows).where(and(eq(periodId), eq(activity)));
    const bySfid = new Map(existing.map(r => [r.sfid, r.id]));
    const incomingSfids = new Set(input.rows.map(r => r.sfid));

    // 2. Insert NEW rows in chunks of 500 (under 65535-param wire cap).
    const toInsert = input.rows.filter(r => !bySfid.has(r.sfid));
    for (const chunk of chunked(toInsert, 500)) await tx.insert(planRows).values(chunk.map(toInsertShape));

    // 3. Update EXISTING rows (per row — small N expected).
    for (const row of input.rows) {
      const id = bySfid.get(row.sfid);
      if (id == null) continue;
      await tx.update(planRows).set(toUpdateShape(row)).where(eq(planRows.id, id));
    }

    // 4. DELETE removed — this is the only branch that can fail with FK RESTRICT.
    const toDeleteIds = existing.filter(r => !incomingSfids.has(r.sfid)).map(r => r.id);
    if (toDeleteIds.length > 0) await tx.delete(planRows).where(inArray(planRows.id, toDeleteIds));

    return { inserted, updated, deleted };
  });
  revalidatePath("/plans");
  return { ok: true, ...result };
} catch (err) {
  if (isFkRestrictError(err)) {
    // Re-query OUTSIDE the failed transaction — tx is rolled back, use the outer `db`.
    const blocked = await queryBlockedDealers(db, periodId, activity, input.rows.map(r => r.sfid));
    return { ok: false, error: `Cannot remove ${blocked.length} dealer(s) with recorded actuals`, blockedDealers: blocked };
  }
  throw err;
}
```

The try/catch is AROUND `db.transaction(...)`, NEVER inside the callback. Order is insert → update → delete because delete is the only failable branch — see PITFALLS.md (RESEARCH "Common Pitfalls" D).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser preview → commitPlanUpload | Untrusted ParsedRow[] crosses into the only writer of plan_rows; the browser can lie about what passed preview. |
| Authenticated session → plan_rows mutation | A request must carry a valid jose session cookie to mutate. |
| App layer → D2-01 atomic-commit invariant | Mirror-semantics commit must be all-or-nothing; partial writes would silently corrupt the allowed-SFID master list. |
| App layer → off-plan guard (D-01) | `executions.plan_row_id ON DELETE RESTRICT` is the structural barrier preventing destructive removal of plan rows with actuals. |
| Dependency supply chain → server bundle | A future `import * as XLSX from "xlsx"` in `lib/actions/*` would re-introduce server-side .xlsx parsing (CVE-2023-30533 surface). |
