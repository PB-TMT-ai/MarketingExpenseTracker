<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Server Action (createPeriod / setActivePeriod) | Untrusted form input (type, label, dates, target period id) crosses into a mutation. |
| Authenticated session → period mutation | A request must carry a valid jose session cookie to mutate periods. |
| App layer → DB single-active invariant | The "exactly one active period" rule (D-11) is an app-enforced integrity constraint inside a transaction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04-01 | Tampering | createPeriod input (label/dates/type) | mitigate | Zod schema validates at action entry: `type` is `z.enum(["month","quarter","fy"])` (rejects arbitrary enum values), `label` non-empty, dates ISO with `endDate >= startDate` refine. Invalid input inserts nothing. |
| T-01-04-02 | Spoofing / Elevation of Privilege | Both Server Actions invoked without auth | mitigate | Each action re-verifies the jose cookie (`verifySession`) before mutating — defense-in-depth beyond proxy.ts (CVE-2025-29927). Unauthorized → throw/redirect, no mutation. |
| T-01-04-03 | Tampering (integrity) | "exactly one active period" race / partial update | mitigate | setActiveTx runs both UPDATEs in a single `db.transaction` (clear-all then set-one), so a concurrent reader never sees two active rows; the live smoke asserts count === 1 after two toggles. |
| T-01-04-04 | Tampering | setActivePeriod(id) targeting a non-existent/foreign id | accept | Worst case: `WHERE id=$1` matches zero rows → zero active periods (recoverable by selecting another); no cross-tenant data (single shared tenant). Low risk; not worth a pre-check round-trip. |
| T-01-04-05 | Information Disclosure | Period data / active-period helper leaking to client | accept | Periods carry no secrets/PII (labels + dates); getActivePeriod() runs server-side. No `process.env` secret is read in any new file. Low risk. |
| T-01-04-SC | Tampering | npm installs | mitigate | This plan installs NOTHING (native inputs + Tailwind only). Acceptance gates assert `package.json` dependencies are unchanged across all three tasks. |

No HIGH-severity threats. Input validation (Zod), auth re-check, and the transactional single-active invariant are the load-bearing mitigations and are each proven by a test or the blocking smoke.
</threat_model>
