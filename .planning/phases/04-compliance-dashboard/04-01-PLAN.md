<objective>
Establish two foundational primitives that the rest of Phase 4 depends on:

1. **Status registry extension.** Introduce `lib/activities/status.ts` exporting `STATUS_VALUES = ['Pending', 'In Progress', 'Done', 'Cancelled'] as const`, `TERMINAL_STATUSES = ['Done', 'Cancelled'] as const`, and `type StatusValue`. Migrate every status-bearing activity config to import this constant so `'Cancelled'` becomes a first-class enum value across the registry. This closes the A4/R1 risk — Phase 3.1 has NOT yet added `'Cancelled'` to the registry, and Phase 4 SQL hardcodes the literal in `filter (where status = 'Cancelled')` clauses.

2. **Shared completeness math.** Introduce `lib/compliance/completeness.ts` implementing `computeCompleteness()` per D-01 / D-02 / D-03 / D-04. This is the single source of truth (D-05) that the dashboard cards, the future grid stat strip, and the Phase 5 export will all import. Asymmetric-denominator spec test ({10,6,2} → {0.75, 0.20}) is mandatory.

Purpose: every downstream Phase 4 task either imports `STATUS_VALUES` / `TERMINAL_STATUSES` or imports `computeCompleteness`. Without this plan, every dashboard SQL string and every percent calc would be a magic constant scattered across the page.
Output: `lib/activities/status.ts`, `lib/compliance/{completeness.ts,completeness.test.ts,index.ts}`, 5 activity configs updated to consume `STATUS_VALUES`.
</objective>
